# План выноса и исправления обувной соты

## Зафиксированные факты

- Размеры соты и посадки в шкаф уже считаются выверенными. Геометрию `a_out`, `b_out`, `r_out`, `cell_len`, толщину стенки и посадочные зазоры не менять без отдельного указания.
- В рабочем каталоге сейчас есть три разные ветки логики соты:
  - `Шкафчик Никиты (Сказка м.г.)/build_cabinet.py` — самая старая встроенная версия, где паз режется простым трапецеидальным профилем, а на наклонных гранях перфорация ограничена тремя линиями.
  - `Шкафчик Никиты (Сказка м.г.)/create_cabinet_model.py` и `build_cabinet.py` в корне — промежуточная версия с отдельным профилем паза, но с очень узкой логикой перфорации только вокруг центра замка.
  - `build_single_cell.py` в корне — наиболее полезная отладочная версия для дальнейшей работы: в ней паз уже отделён в собственный профиль, а перфорация размещается по всей грани в шахматном порядке с исключением центральной зоны у паза.
- В `scratch/check_overlap.py` уже есть полезная регрессия на стыковку соседних ячеек. Текущий результат: пересечение равно нулю.
- Запущенный `FreeCADCmd.exe` сейчас исполняет именно `c:\3DModel\build_single_cell.py`, а не `.pyc`. Выходной файл `SingleCell.FCStd` не создан, значит процесс не дошёл до сохранения.
- Текущая версия `build_single_cell.py` пытается вырезать около 648 loft-каттеров перфорации одним compound boolean. Это наиболее вероятная причина зависания и роста памяти выше 10 ГБ.

## Подтверждённые проблемы

### 1. Дефект скругления у паза

- Встроенная в шкаф версия режет паз через `make_centered_trapezoid_face(...)` с параметрами `0.4, -3.2, 24.3, 30.7, 0.4, 0.0`.
- Такой профиль геометрически слишком близок к старой логике шипа и не имеет отдельной формы входной воронки. Из-за этого привязка скругления у входа в паз получается хрупкой и визуально даёт дефект на входной кромке.
- Более поздняя логика из `build_single_cell.py` и промежуточной версии уже ушла от этого к отдельной функции `make_female_cutter_face(...)`, где входные углы задаются как отдельные вершины профиля, а не просто радиусы на трапеции.

### 2. Неполная перфорация на пазе и гребне

- В версии, встроенной в шкаф, наклонные грани получают только `[-23.5, 0.0, 23.5]` или `[-25.0, 0.0, 25.0]`. Это даёт всего по одной линии по центру и по одной с каждой стороны, а не максимальное шахматное заполнение грани.
- В промежуточной версии из `create_cabinet_model.py` проблема другая: там отверстия на наклонных гранях вообще сведены к центральной зоне замка `[-3.5, 3.5]` и `[-7.0, 0.0, 7.0]`.
- В `build_single_cell.py` логика уже ближе к требованию: отверстия раскладываются по всей длине грани, но у пазовых граней вводится ограничение `PERF_GROOVE_MIN`, чтобы не ослабить паз.

### 3. Непригодный режим отладки

- Отладочный скрипт строит правильнее, но слишком тяжело режет перфорацию: один compound из сотен loft-каттеров.
- Для FreeCAD это плохой режим именно для разработки. Он затрудняет локальную отладку паза и перфорации, потому что каждое изменение упирается в долгий boolean-cut.

## План выноса в отдельный проект

### Цель

Выделить соту в отдельный мини-проект, где шкаф больше не участвует в построении геометрии, а только импортирует готовую форму или вызывает тонкий адаптер сборки.

### Рекомендуемая структура

```text
Шкафчик Никиты (Сказка м.г.)/
  honeycomb_cell/
    README.md
    config.py
    build_single_cell.py
    export_single_cell.py
    geometry/
      profile.py
      joints.py
      perforation.py
      fillets.py
    tests/
      check_overlap.py
      inspect_groove.py
      inspect_perforation.py
```

### Этап 1. Заморозить параметры

- Собрать все размеры соты, замков, перфорации и торцевых скруглений в один конфиг.
- Отдельно отметить в README, что параметры размеров запрещено менять без явной команды.
- Сразу разделить параметры на две группы:
  - неизменяемые посадочные размеры;
  - отладочные флаги построения: без перфорации, только паз, только шип, staged boolean.

### Этап 2. Взять `build_single_cell.py` как базу, а не встроенный шкафный скрипт

- Эта версия уже ближе всего к нужной постановке задачи по перфорации.
- Из неё нужно забрать:
  - отдельный профиль female-паза;
  - полную раскладку отверстий по всей грани;
  - ограничение центральной зоны только для пазовых граней.
- Из шкафного скрипта брать только размещение пяти ячеек внутри секции и экспорт сборки.

### Этап 3. Разделить геометрию на независимые стадии

- Стадия A: внешний и внутренний контур + базовая полая оболочка.
- Стадия B: торцевые скругления.
- Стадия C: усилительные площадки.
- Стадия D: шипы.
- Стадия E: пазы.
- Стадия F: перфорация.

Это нужно не ради красоты, а чтобы каждый дефект можно было смотреть отдельно, без полной пересборки всей модели.

### Этап 4. Исправить паз на базе отдельного female-профиля

- Перенести в отдельный модуль реализацию `make_female_cutter_face(...)`.
- Сначала собрать одиночную ячейку в режиме:
  - без перфорации;
  - только один паз;
  - без размещения в шкафу.
- Проверить, что входное скругление формируется именно профилем паза, а не случайной комбинацией радиусов трапеции.
- Если дефект останется, отлаживать уже только локальный профиль и его привязку к середине грани, не трогая размеры.

### Этап 5. Перенести шахматную перфорацию в отдельный генератор размещения

- Сначала генератор должен возвращать только список позиций отверстий по граням, без резки тела.
- Для каждой грани нужно отдельно считать:
  - допустимую полуширину;
  - минимальный отступ от торцов;
  - запретную центральную зону у паза;
  - режим гребня и обычной стенки.
- После этого только отдельным шагом превращать позиции в каттеры.
- Цель: добиться максимального числа линий на гребне и пазе в шахматном порядке, но с сохранением минимального запаса до кромок и зоны замка.

### Этап 6. Убрать один гигантский boolean-cut

- Не вырезать сразу весь compound из 648 каттеров.
- Вместо этого использовать один из режимов:
  - резка по одной грани;
  - резка по нескольким рядам;
  - экспорт отладочной модели без перфорации и отдельной модели только с перфорацией;
  - переключатель `ENABLE_PERFORATION_PREVIEW` для проверки раскладки без boolean.
- Это критично, иначе отдельный проект всё равно будет плохо отлаживаться.

### Этап 7. Оставить шкаф только потребителем готовой ячейки

- В шкафном проекте должен остаться тонкий адаптер:
  - либо импорт готового STEP/FCStd одной ячейки;
  - либо вызов функции `create_honeycomb_cell_shape()` из отдельного модуля.
- Логика паза, шипа, перфорации и торцевых скруглений должна жить только в одном месте.
- Это уберёт текущее расхождение между `build_cabinet.py`, `create_cabinet_model.py` и `build_single_cell.py`.

## Минимальный набор проверок после выноса

- Ячейка собирается без шкафа и сохраняется в отдельный файл.
- Паз визуально чистый на входной кромке.
- На пазе и гребне есть максимальное число линий перфорации в шахматном порядке, но без выхода к краям.
- `scratch/check_overlap.py` или его копия в `tests/` подтверждает нулевое пересечение соседних ячеек.
- Пересборка одиночной ячейки проходит без зависания `FreeCADCmd.exe` и без роста памяти до десятков гигабайт.

## Практический порядок следующей реализации

1. Создать отдельную папку `honeycomb_cell` и перенести туда именно `build_single_cell.py` как базовый файл.
2. Вынести параметры в конфиг и добавить режимы `NO_PERF`, `ONLY_GROOVE`, `ONLY_RIDGE`, `PERF_PREVIEW`.
3. Добить чистый паз без перфорации.
4. После этого вернуть шахматную перфорацию и перевести boolean-cut на staged-режим.
5. Только потом обратно подключать соту в сборку шкафа.
